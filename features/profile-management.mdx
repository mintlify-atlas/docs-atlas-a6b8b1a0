## Overview

The profile management system provides a centralized dashboard to view scraped accounts, monitor your username pool status, and track assignment readiness. All profiles are automatically deduplicated and organized by job for multi-tenant isolation.

## Extracted Accounts Table

The main interface for viewing scraped profiles is the **Extracted Accounts** table, powered by the `PaymentsTable` component.

Location: `components/payments-table.tsx`

### Features

- **Paginated Display**: Shows 10 accounts per page with Previous/Next navigation
- **Profile Details**: Displays ID, full name, and username for each account
- **Real-time Updates**: Automatically refreshes after scraping campaigns
- **Assignment Actions**: Quick access to VA distribution workflows

## Database Schema

### global_usernames Table

The primary table for deduplicated username storage:

```sql
CREATE TABLE global_usernames (
  username TEXT PRIMARY KEY,
  used BOOLEAN DEFAULT FALSE,
  base_id TEXT NOT NULL,
  job_id UUID,
  created_at TIMESTAMP DEFAULT NOW()
);
```

**Columns Explained**:

- `username`: Unique username (enforced at database level)
- `used`: Boolean flag - `false` = available, `true` = assigned to a campaign
- `base_id`: Multi-tenant isolation (Airtable base ID)
- `job_id`: Reference to the scraping job that added this username
- `created_at`: Timestamp when username was first scraped

### scrape_results Table

Stores full profile metadata:

```sql
CREATE TABLE scrape_results (
  id TEXT PRIMARY KEY,
  username TEXT,
  full_name TEXT,
  follower_count INTEGER,
  following_count INTEGER,
  post_count INTEGER,
  is_verified BOOLEAN,
  is_private BOOLEAN,
  biography TEXT,
  url TEXT,
  detected_gender TEXT,
  base_id TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);
```

**Rich Metadata**:
- Social proof metrics (followers, following, posts)
- Verification status
- Account privacy level
- Profile bio for targeting analysis
- Gender detection results

## Profile Lifecycle

Understanding how profiles move through the system:

<Steps>
  <Step title="Scraping">
    Profiles are extracted from source accounts via Apify and saved to `scrape_results`.
  </Step>
  
  <Step title="Deduplication">
    Usernames are added to `global_usernames` with `used = false`.
    
    Duplicates are automatically rejected by the PRIMARY KEY constraint.
  </Step>
  
  <Step title="Available Pool">
    Profiles remain in the unused pool until assigned to a campaign.
    
    Query: `SELECT COUNT(*) FROM global_usernames WHERE used = false AND base_id = ?`
  </Step>
  
  <Step title="Campaign Assignment">
    When a campaign is created, profiles are selected and marked `used = true`.
    
    This prevents the same profile from being assigned twice.
  </Step>
  
  <Step title="VA Distribution">
    Assigned profiles are distributed across VA tables and synced to Airtable.
  </Step>
  
  <Step title="Cleanup (7 days)">
    After 7 days, old campaigns are cleaned up and profiles are marked `used = false` again, returning them to the available pool.
  </Step>
</Steps>

## Viewing Profile Statistics

### Per-Job Statistics

Location: `lib/scraping-jobs.ts` → `getJobStatisticsByBaseId()`

**Metrics Available**:
- `totalUsernames`: All profiles in the database for this job
- `usedUsernames`: Profiles currently assigned to active campaigns
- `availableUsernames`: Profiles ready to be assigned
- `totalAssignments`: Total profiles distributed to VAs historically
- `totalScrapeResults`: Raw scraping results count

### Real-time Queries

The platform uses Supabase real-time subscriptions for instant updates:

```javascript
// Available usernames count
const { count } = await supabase
  .from('global_usernames')
  .select('*', { count: 'exact', head: true })
  .eq('base_id', baseId)
  .eq('used', false)
```

## Multi-Tenant Isolation

All profile queries are scoped to the active job using `base_id`:

<Warning>
  **Critical**: Never query profiles without filtering by `base_id`. This ensures clients only see their own data in a multi-tenant environment.
</Warning>

### Context-Aware Client

```javascript
import { createSupabaseClientWithContext } from '@/lib/supabase'

// Always pass base_id for tenant isolation
const supabaseWithContext = createSupabaseClientWithContext(baseId)

const { data } = await supabaseWithContext
  .from('global_usernames')
  .select('*')
  .eq('base_id', baseId) // Belt-and-suspenders filtering
  .eq('used', false)
```

## Assignment Readiness Checks

Before assigning profiles to VAs, the system validates availability:

### Fixed 180 Assignment

**Required Profiles**: 14,400 (180 profiles × 80 VAs)

**Validation Flow**:
1. User clicks "Assign 180 accounts per VA"
2. System queries unused username count for current `base_id`
3. If `count >= 14400`: ✅ Proceed to campaign creation
4. If `count < 14400`: ❌ Show error toast

```javascript
const { count } = await supabase
  .from('global_usernames')
  .select('*', { count: 'exact', head: true })
  .eq('base_id', baseId)
  .eq('used', false)

if (count < 14400) {
  toast({
    title: "Not enough profiles",
    description: "Please scrape more profiles before distributing.",
    variant: "destructive"
  })
}
```

### Custom Assignment

**User Configurable**: Specify profiles per VA (e.g., 50, 100, 250)

**Required Profiles**: `profilesPerVA × 80`

**Validation**: Checked during campaign creation (backend validation)

## Profile Data Retention

The platform implements a 7-day cleanup cycle:

### Cleanup Workflow

**Trigger**: Daily cron job at 2 AM

**Endpoint**: `POST /api/cleanup`

**Actions**:
1. Identify campaigns older than 7 days
2. Mark all profiles from those campaigns as `used = false`
3. Delete campaign records
4. Free up profiles for reassignment

**Cron Configuration**:
```bash
# Add to crontab
0 2 * * * curl -X POST http://localhost:5001/api/cleanup
```

<Note>
  This allows profiles to be recycled after VAs complete outreach, maximizing your database efficiency.
</Note>

## Troubleshooting

### Profiles Not Appearing

**Symptoms**: Table shows "No accounts to display" after scraping

**Solutions**:
1. Check if scraping completed successfully (progress bar reached 100%)
2. Verify gender filter didn't exclude all results
3. Check browser console for JavaScript errors
4. Reload the page to refresh data

### Duplicate Profiles

**Symptoms**: Same username appearing multiple times

**Cause**: Database constraint violation or RLS policy misconfiguration

**Solutions**:
1. Check `global_usernames` table has PRIMARY KEY on `username`
2. Run deduplication query:
   ```sql
   DELETE FROM global_usernames a USING global_usernames b
   WHERE a.ctid < b.ctid AND a.username = b.username;
   ```

### "Not enough profiles" Error

**Symptoms**: Cannot assign to VAs despite having scraped profiles

**Cause**: Profiles marked as `used = true` from previous campaigns

**Solutions**:
1. Run cleanup endpoint: `POST /api/cleanup`
2. Manually reset profiles:
   ```sql
   UPDATE global_usernames 
   SET used = false 
   WHERE base_id = 'your-base-id' AND used = true;
   ```
3. Scrape more profiles to reach the 14,400 threshold
